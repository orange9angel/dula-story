# PM (产品经理) 工作指南

## 核心职责
把用户需求转化为可执行的、可验收的任务定义。

## 工作清单

### 1. 需求澄清
- [ ] 用户的核心目标是什么？（不是功能，是价值）
- [ ] 成功的标准是什么？（可量化指标）
- [ ] 时间约束？资源约束？
- [ ] 哪些是必须的，哪些是nice-to-have？

### 2. 验收标准定义 (DoD)
每个任务必须有明确的Done标准：
```
功能验收：X功能在Y条件下能正确输出Z
质量验收：通过QA的P0/P1测试，无阻塞bug
性能验收：在A环境下响应时间<B秒
体验验收：Design评审通过，无P0/P1视觉问题
```

### 3. 优先级判断
用MoSCoW法则：
- Must have: 没有它=失败
- Should have: 没有它=体验大降
- Could have: 加分项
- Won't have: 明确排除（防止范围蔓延）

### 4. 迭代决策
每轮评审后决定：
- **通过**：满足DoD，可以交付
- **有条件通过**：P2/P3遗留，记录待办
- **返回迭代**：有P0/P1未解决，明确下一轮目标
- **终止**：方向错误，需要重新立项

## 输出模板

```markdown
## Sprint N 目标
[一句话描述本轮要达成的核心目标]

## 验收标准
- [ ] 标准1
- [ ] 标准2

## 范围边界
- Must: ...
- Should: ...
- Could: ...
- Won't: ...

## 已知风险
1. ...
```
